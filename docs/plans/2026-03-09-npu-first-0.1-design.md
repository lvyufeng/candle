# Candle 0.1 NPU-First Design

## Context

Candle is migrated from MindTorch v2 and currently exposes a broad torch-like API surface. For 0.1, we choose a focused NPU-first release strategy rather than broad feature coverage. The release target is usability and stability for a single-card Ascend 910B training path.

## Confirmed Product Scope

### Success Definition (0.1)

- Primary goal: single-card NPU training is stable and repeatable.
- Required baseline: one minimal end-to-end training loop must pass consistently.
- Hardware baseline: Ascend 910B single card.
- Secondary requirement: CPU fallback path must remain CI green as development safety net.

### Backend Positioning

- GA (0.1): NPU single-card training core path + CPU fallback baseline.
- Experimental: selected advanced NPU ops and compatibility runners.
- Not supported in 0.1 scope: distributed high-level features (FSDP/DeviceMesh/Tensor Parallel), true JIT/compile acceleration, full ONNX export stack.

## Architectural Direction

### 1) Capability Surface Must Match Validation Surface

The externally documented capability matrix must match what CI/release gates actually validate. For 0.1:

- Reduce ambiguity in README and docs by explicitly marking supported vs experimental vs unsupported.
- Keep broad API presence only when behavior is intentionally defined (including explicit not-supported errors).
- Avoid implicit promises through module exposure alone.

### 2) Golden Training Path

Define a single deterministic golden path:

- Data: synthetic fixed-shape data.
- Model: minimal trainable module (e.g., MLP/Linear stack).
- Loop: forward -> loss -> backward -> optimizer.step -> checkpoint save/load -> continue.
- Determinism controls: fixed seed, fixed step count, fixed dtype/device.
- Output artifacts: loss trace, NaN/Inf checks, per-step timing summary.

The golden path is the release-critical runtime contract.

### 3) Error Context Normalization

All NPU critical-path failures should include stable context fields:

- `op`, `device`, `dtype`, `shape`, `seed`, `step`, and where relevant backend stage.

This enables direct reproduction and reduces triage latency in CI and field reports.

## Testing and Release Gates

### Gate Layer A: PR Fast Gate

- CPU contract + CPU core tests.
- NPU smoke subset when NPU runner available.
- Goal: fast feedback, prevent obvious regressions.

### Gate Layer B: Main Stability Gate

- NPU golden training loop.
- Serialization continuity test (save/load + continue train).
- Key operator regression subset tied to golden path.

### Gate Layer C: Release Gate

- Repeat NPU golden path multiple runs on 910B.
- Require zero flaky failures.
- Attach artifacts: environment summary, loss curve JSON/CSV, failure context logs.

Release is blocked if any P0 issue remains unresolved or gate criteria are not met.

## Milestones (2-3 Weeks)

### M1: Scope Freeze and Baseline Setup

- Publish support matrix (GA/Experimental/Not supported).
- Align README and release messaging.
- Add golden path script + first deterministic baseline result.

### M2: Core Path Hardening

- Close high-impact gaps on the golden path.
- Add explicit unsupported semantics for out-of-scope high-level features.
- Enforce standardized error context in critical modules.

### M3: Release Process Hardening

- Integrate NPU release gate job.
- Add pre-release verification checklist and artifact upload.
- Dry-run release workflow before tagging.

## Risk Controls

- Scope creep control: non-critical enhancements moved to 0.2 backlog.
- Stability-first policy: favor predictable behavior over broad API expansion.
- Evidence-first release: no release claims without passing gate artifacts.

## Deliverables for 0.1

- A documented support matrix that matches behavior and CI.
- A deterministic and repeatable 910B single-card training baseline.
- A release workflow with mandatory NPU-first verification gates.
- Clear unsupported/experimental boundaries to protect user expectations.
